# StoryLearn AI backend

FastAPI service that turns a topic + age group into an illustrated story (chapter count scales
with age group) with 2 quiz questions per chapter, via a per-chapter agent pipeline, and caches
the result in Redis.

## Setup

```
cd server
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env   # fill in ANTHROPIC_API_KEY, IMAGE_API_KEY (OpenAI) at minimum
uvicorn main:app --reload --port 8000
```

Requires a local Redis instance for caching (`redis-server`, or `docker run -p 6379:6379 redis`).

## Pipeline (`pipeline.py`)

```
topic, age_group
    -> plan_chapters()              -> Nx ChapterOutline{title, research_query}   (N varies by age group)
    -> research(research_query)     -> ResearchFacts   (sequential, once per chapter)
    -> write_chapter() + write_quiz_for_chapter()   -> (concurrent across all N chapters)
    -> generate_image()             -> attaches image_url
    -> StoryPayload                 -> cached in Redis, returned to the frontend
```

1. **Plan** (`agents/chapter_planner.py`) — one Claude call splits the topic into chapters, each
   with a display `title` and a separate plain `research_query`. Chapter count scales with age
   group via `agents/_age_group_style.CHAPTER_COUNT_RANGE` — Claude picks within that range based
   on topic depth, it's not a fixed number:

   | Age group | Chapters |
   |---|---|
   | elementary | 3 |
   | middle_school | 4-5 |
   | high_school | 6-8 |
   | college | 8-10 |

2. **Research** (`agents/researcher.py`) — the "mega-crawler": one continuous Stagehand/
   Browserbase session per chapter (Wikipedia summary, must succeed; then Google search for
   2 articles + 2 YouTube videos, each independently best-effort). Runs sequentially, one
   chapter's research at a time (up to 10x for college), since a single Browserbase plan may
   only allow one concurrent session. Falls back to mock data if `BROWSERBASE_API_KEY`/
   `BROWSERBASE_PROJECT_ID` are unset or a chapter's Wikipedia step fails. Note: for the larger
   chapter counts this can take several minutes total — not yet optimized, accepted for now.
3. **Write** (`agents/storyteller.py`, `agents/quiz_master.py`) — once all chapters' research
   is done, these run concurrently across chapters via `asyncio.gather`. Storyteller writes one
   chapter's text + image prompt; Quiz Master then writes that chapter's 2 quiz questions (each
   with an `explanation`), grounded only in the finished chapter text (never the raw research).
   `chapter_index` is not generated by Quiz Master — `pipeline.py` assigns it afterward based on
   which chapter's call produced the question. Total quiz length is always 2x the chapter count.
4. **Images** (`agents/images.py`) — calls OpenAI's `gpt-image-1` per chapter's image prompt,
   with a 60s timeout, one retry on transient failure, and a placeholder fallback so a slow/
   failed image never hangs the pipeline. Returns a base64 data URI.
5. **Cache** (`cache.py`) — Redis get/set keyed by `storystream:{topic-slug}:{age_group}`, plus
   a capped, deduplicated `LPUSH`/`LTRIM` history list backing `GET /api/history`.

All Claude calls (`chapter_planner`, `storyteller`, `quiz_master`) use forced tool-use
(`tool_choice`), not prose-JSON parsing — see `agents/_tool_schemas.py`.

## Layout

- `main.py` - FastAPI app: `POST /api/generate`, `GET /api/history` (fails open to `[]` if Redis is unreachable)
- `pipeline.py` - the orchestrator above
- `agents/chapter_planner.py` - splits the topic into chapter outlines (count scales by age group)
- `agents/researcher.py` - mega-crawler (Wikipedia + Google + articles/YouTube), real Stagehand calls with mock fallback
- `agents/storyteller.py` - writes one chapter's text + image prompt at a time
- `agents/quiz_master.py` - writes one chapter's 2 quiz questions (+ explanations) at a time
- `agents/images.py` - OpenAI `gpt-image-1` image generation, owned by the image/integrations teammate
- `agents/_tool_schemas.py`, `agents/_age_group_style.py` - shared helpers (tool-use JSON schemas, age-group tone/style dicts)
- `cache.py` - Redis get/set, plus the history list
- `models.py` - shared Pydantic schemas (source of truth for the wire contract). `AgeGroup` is
  `elementary | middle_school | high_school | college`, matching the frontend exactly.
- `sentry_init.py` - no-op unless `SENTRY_DSN` is set

## Known gaps

- **BAND platform**: still stubbed (env vars only, no real calls) — see `pipeline.py`'s
  module docstring for the intended integration seam once real credentials exist.
- **`GET /api/history` ordering**: most-recent-first via a Redis list, capped at 50 entries and
  deduplicated by `(topic, age_group)` — there's no per-entry timestamp, so it can't show *when*
  a topic was generated, only that it was.
